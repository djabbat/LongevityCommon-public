# Code Audit — MitoROS

## ✅ Соответствия

1.  **Параметры:** Все параметры, перечисленные в `PARAMETERS.md`, присутствуют в коде в структуре `CounterParams` (`src/lib.rs`). Базовая логика кинетического уравнения реализована.
2.  **Формулы:** Формула в коде (`compute_damage`) точно соответствует каноническому виду из `CONCEPT.md`:
    *   `D = d0 + alpha * (n / n_star) + beta * (t / tau_days) + gamma * coupling`
3.  **Канонические определения:** Код строго соблюдает канон `CORRECTIONS_2026-04-22`:
    *   **γ_i = 0:** Параметр `gamma` по умолчанию установлен в `0.0` (`CounterParams::default()`).
    *   **Нет устаревших утверждений:** В коде (`.rs` файлы) полностью отсутствуют упоминания:
        *   `Health Score`
        *   `"MCOA Test 2"` как источника параметров
        *   `χ_Ze` и любые утверждения о его валидации (`R²=0.84`, `"validated biomarker"`)
    *   **Нет устаревшей структуры EIC:** В коде нет упоминаний `Ze`, `BioSense`, `Aqtivirebuli` как рабочих пакетов.
4.  **Consistency (Логика):**
    *   Логика вычисления критического порога (`is_above_critical`, `divisions_to_critical`) соответствует концепции `d_critical` из `PARAMETERS.md`.
    *   Тканеспецифичные веса `w_i(tissue)` заданы `a priori` в `src/tissue.rs` (`tissue.weight()`), что соответствует аксиоме M3 MCOA, описанной в `CONCEPT.md`.

## ❌ Проблемы

1.  **Несоответствие значений параметров по умолчанию между кодом и документацией.**
    *   **Файл:** `src/lib.rs` (строки 46-54, `CounterParams::default`)
    *   **Проблема:** Значения параметров по умолчанию в коде **не соответствуют** гипотетическим/оценочным значениям, указанным в `PARAMETERS.md`. Это критическое несоответствие для воспроизводимости и понимания модели.
        *   `alpha`: код = `0.0000`, док = `0.001` (TBD, гипотетическое)
        *   `beta`: код = `0.5000`, док = `0.01 год⁻¹` (TBD, гипотетическое)
        *   `n_star`: код = `100.00`, док = `10² – 10⁴` (диапазон, TBD)
        *   `tau_days`: код = `29200.0` (~80 лет), док = `10 – 50 лет` (диапазон, TBD)
        *   `d_critical`: код = `0.5000`, док = не указано единое значение (контекстно)
    *   **Риск:** Калькулятор (`main.rs`) и интеграционные тесты используют эти некорректные по умолчанию значения, что делает их выводы несопоставимыми с концептуальной моделью.

2.  **Наличие устаревших/отозванных упоминаний в core-документах.**
    *   **Файл:** `CONCEPT.md` (строка 3 в примечании, строка 92)
    *   **Проблема:** В документации сохранены ссылки на отозванные или переосмысленные концепции:
        *   Строка 3: `> ⚠️ **См. [../CORRECTIONS_2026-04-22.md](../CORRECTIONS_2026-04-22.md)** — некоторые утверждения могут быть отозваны.` — Формулировка "могут быть" ослабляет обязательность применения канона. Согласно `CORRECTIONS`, он является **единственным источником истины**.
        *   Строка 92 (и другие): Упоминания `Ze` в контексте "*Ze Vector Theory*" допустимы как теоретические, но **необходимо явно указать**, что `Ze` более не является частью структуры EIC (как было отозвано в §1.4 CORRECTIONS). В текущем виде это может вводить в заблуждение о статусе проекта.

3.  **Неполная реализация концепции взаимодействий (Γ-матрица).**
    *   **Файл:** `src/lib.rs`, `src/trajectory.rs`
    *   **Проблема:** В `PARAMETERS.md` декларируется связь `Γ_{3,4}` (Epigenetic Drift → MitoROS) и общая структура `γ_3 · I = Σ Γ_{3,j} · D_j`. Однако в коде `coupling` представлен как единый скаляр `f64`, поступающий извне. **Отсутствует внутренняя модель расчета влияния других счетчиков**, что делает параметр `gamma` нефункциональным без внешнего управления. Код не отражает заявленную в документах сложность взаимодействий.

## ⚠️ Улучшения

1.  **Валидация параметров:** Метод `CounterParams::validate()` можно расширить для проверки не только на `>0`, но и на соответствие физиологически правдоподобным диапазонам (например, `tau_days` от 3650 до 18250 для диапазона 10-50 лет).
2.  **Документация кода:** В комментариях к `CounterParams` и `compute_damage` можно добавить прямые ссылки на уравнения из `CONCEPT.md` и `PARAMETERS.md` для улучшения навигации.
3.  **Тестовые данные:** Интеграционные тесты (`integration.rs`) используют произвольные `division_rate_per_day`. Хорошей практикой было бы добавить тесты, основанные на конкретных анкорах из литературы, аналогично скрипту `calibrate.py`.
