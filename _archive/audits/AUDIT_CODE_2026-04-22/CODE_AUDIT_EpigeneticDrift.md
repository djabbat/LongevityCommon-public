# Code Audit — EpigeneticDrift

## ✅ Соответствия

1.  **Параметры (базовый набор):** В коде (`src/lib.rs`) присутствуют все ключевые параметры из уравнения в CONCEPT.md и PARAMETERS.md: `d0`, `alpha`, `beta`, `gamma`, `n_star`, `tau_days`. Параметр `d_critical` логично добавлен для определения критического порога.
2.  **Формула:** Кинетическое уравнение `D₄(n, t) = D₄,₀ + α₄·(n / n₄*) + β₄·(t / τ₄) + γ₄·I(others)` из CONCEPT.md корректно реализовано в функции `compute_damage` (`src/lib.rs:76`).
3.  **Канонические определения (CORRECTIONS_2026-04-22):**
    *   ✅ **Health Score:** Упоминания формулы Health Score или "4 домена" в предоставленных файлах кода и core-документах **отсутствуют**.
    *   ✅ **MCOA Test 2:** В коде и документах **нет ссылок** на "MCOA Test 2" как источник параметров.
    *   ✅ **χ_Ze:** Упоминаний χ_Ze как биомаркера с R²=0.84 или "validated" в коде и документах EpigeneticDrift **нет**.
    *   ✅ **γ_i = 0 by default:** Параметр `gamma` в структуре `CounterParams` инициализирован значением `0.0` по умолчанию (`src/lib.rs:48`), что соответствует канону.
    *   ✅ **Согласованность v:** Формула `v = N_S/(N-1)` в этом подпроекте не используется.
4.  **Consistency (логика):** Логика кода соответствует концепции разделения вклада времени (`beta`) и делений (`alpha`). В модуле `tissue` (`src/tissue.rs:36-50`) эвристически корректируются `alpha` и `beta` для постмитотических (нейрон, кардиомиоцит) и пролиферирующих (крипта кишечника) тканей, что отражает утверждения из CONCEPT.md.
5.  **Устаревшие упоминания:** В предоставленных файлах кода и документах проекта EpigeneticDrift **нет упоминаний** Ze, BioSense, Aqtivirebuli как WP EIC, а также нет утверждений о χ_Ze как биомаркере или Health Score.

## ❌ Проблемы

1.  **Критическое несоответствие параметра `tau` (характерное время).**
    *   **CONCEPT.md / PARAMETERS.md:** Указано ориентировочное значение `τ₄ = 10.0` лет, основанное на эпидемиологических данных (Horvath clock, DunedinPACE).
    *   **Код (`src/lib.rs:52`):** В реализации по умолчанию `tau_days = 36500.0` (что эквивалентно **100 годам**). Это на порядок отличается от документации и делает вклад временного слагаемого `beta * (t / tau)` незначительным на масштабах человеческой жизни.
    *   **Скрипт `calibrate.py` (строка 12):** Использует `TAU_YEARS = 100.0` для калибровки, усиливая это несоответствие.

2.  **Несоответствие параметра `n_star` (характерное число делений).**
    *   **PARAMETERS.md:** Указано значение `n₄* = 50` (гипотетически для стволовых клеток).
    *   **Код (`src/lib.rs:51`):** Используется `n_star = 100.00`.
    *   **Скрипт `calibrate.py` (строка 11):** Использует `N_STAR_DIV = 100.0`.

3.  **Упрощённая модель связи (`gamma`).** В коде параметр `gamma` — один скаляр. В PARAMETERS.md описаны разные коэффициенты связи с другими счётчиками (`γ₄,₁`, `γ₄,₂`, `γ₄,₃`, `γ₄,₅`, `γ₄,₄`). Код не поддерживает эту детализацию, что ограничивает возможность моделирования специфических взаимодействий.

4.  **Отсутствует интеграция с MCOA (веса тканей).**
    *   **CONCEPT.md:** Упоминается, что тканеспецифический возраст `L_tissue` рассчитывается как взвешенная сумма состояний счётчиков, где `w₄(tissue)` — вес эпигенетического дрейфа.
    *   **Код (`src/tissue.rs`):** Хотя функция `weight()` существует, **она нигде не используется** в расчётах ущерба (`compute_damage`) или в интеграции с другими счётчиками. Проект изолирован.

5.  **Некорректная калибровка.** Скрипт `calibrate.py` использует синтетические "якорные" точки `[(0, 0, 0.0), (0, 25, 0.25), ...]`, не привязанные к реальным данным, что противоречит духу проекта, основанного на литературе. Параметры `TAU_YEARS` и `N_STAR_DIV` в нём также противоречат документации.

## ⚠️ Улучшения

1.  **Структура параметров связи.** Для будущей расширяемости стоит заменить скаляр `gamma` на структуру или enum, позволяющую задавать влияние других конкретных счётчиков (например, `coupling_mitoros: f64`).
2.  **Валидация параметров.** Текущая `validate()` проверяет только неотрицательность. Стоит добавить проверки на осмысленные физиологические диапазоны (например, `tau_days` от 5 до 30 лет в переводе в дни).
3.  **Документация кода.** В комментариях к модулям и функциям можно добавить прямые ссылки на соответствующие разделы `CONCEPT.md` и `PARAMETERS.md` для улучшения прослеживаемости.
4.  **Тестовые данные.** Интеграционные тесты используют абстрактные параметры. Хорошо бы добавить модульные тесты, проверяющие поведение модели на реальных литературных данных (например, ожидаемый `D` для определённого возраста в конкретной ткани).
