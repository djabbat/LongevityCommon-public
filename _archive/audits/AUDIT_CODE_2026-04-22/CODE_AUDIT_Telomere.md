# Code Audit — Telomere

## ✅ Соответствия

1.  **Формулы:** Кинетическое уравнение в коде (`compute_damage`) в точности соответствует уравнению в `CONCEPT.md`:
    *   `D₂(n, t) = D₂,₀ + α₂·(n / n₂*) + β₂·(t / τ₂) + γ₂·I(others)` (CONCEPT.md, Раздел 3)
    *   `p.d0 + p.alpha * (n / p.n_star) + p.beta * (t_days / p.tau_days) + p.gamma * coupling` (`src/lib.rs`, строка 60)

2.  **Параметры (основные):** Все базовые параметры из `PARAMETERS.md` присутствуют в структуре `CounterParams`:
    *   `D₂,₀` -> `d0`
    *   `α₂` -> `alpha`
    *   `β₂` -> `beta`
    *   `n₂*` -> `n_star`
    *   `τ₂` -> `tau_days`
    *   Каноническое `γ_i = 0` строго соблюдается: `gamma` по умолчанию `0.0`.

3.  **Логика и согласованность:**
    *   Реализована тканеспецифичная корректировка параметров (`src/tissue.rs`), что соответствует обсуждению тканевой специфичности в `CONCEPT.md`.
    *   Весовой коэффициент `w₂(tissue)` определен в коде (`Tissue::weight()`), что соответствует MCOA-рамке. Он не используется внутри расчета `damage`, что верно, так как вес применяется на уровне агрегации (`L_tissue`), а не самого счетчика.
    *   В коде и документах **отсутствуют** прямые упоминания отозванных концепций: `Health Score`, `"MCOA Test 2"`, `χ_Ze` как валидированный биомаркер, а также устаревшая структура EIC WP (Ze/BioSense/Aqtivirebuli).

## ❌ Проблемы

1.  **Расхождение в значениях `τ₂`:** В `PARAMETERS.md` для параметра `τ₂` указан статус **"TBD (to-be-determined)"**, однако в коде (`src/lib.rs`, строка 33) установлено жесткое значение по умолчанию `tau_days: 32850.0` (~90 лет). Это прямое противоречие между документацией и реализацией.
    *   **Файл:** `crates/telomere_counter/src/lib.rs`
    *   **Строка:** 33 (в `impl Default for CounterParams`)
    *   **Несоответствие:** Код присваивает параметру конкретное значение, в то время как `PARAMETERS.md` объявляет его неизмеренным.

2.  **Неполное отображение параметров связи:** `PARAMETERS.md` детализирует гипотетические коэффициенты связи `Γ_{2,1}`, `Γ_{2,3}`, `Γ_{2,4}`, `Γ_{2,5}` с другими счетчиками. В коде же существует только один обобщенный параметр `gamma`. Это упрощение мешает экспериментальной проверке конкретных гипотез о связях, описанных в `CONCEPT.md` и `KNOWLEDGE.md`.
    *   **Файл:** `crates/telomere_counter/src/lib.rs` (структура `CounterParams`)
    *   **Проблема:** Архитектура кода не отражает научную детализацию, заложенную в документах.

3.  **Отсутствие параметра `d_critical` в документации:** В коде критический порог `d_critical` является ключевым параметром (используется в `is_above_critical`, `divisions_to_critical`). Однако в `PARAMETERS.md` этот параметр отсутствует, что создает пробел в документации.
    *   **Файл:** `crates/telomere_counter/src/lib.rs`
    *   **Проблема:** Документация не описывает все параметры, используемые в модели.

4.  **Несоответствие единиц измерения и нормализации:** `PARAMETERS.md` приводит параметры `α₂` и `β₂` в размерных единицах (bp/PD, bp/year), в то время как в коде `alpha` и `beta` — безразмерные коэффициенты. Это создает риск неправильной интерпретации и калибровки. Скрипт `calibrate.py` работает с безразмерными величинами, что усиливает путаницу.
    *   **Файлы:** `PARAMETERS.md`, `crates/telomere_counter/src/lib.rs`, `scripts/calibrate.py`
    *   **Проблема:** Отсутствует явное описание процедуры нормализации, связывающей литературные данные (в bp) с внутренними безразмерными параметрами кода.

## ⚠️ Улучшения

1.  **Верификация входных данных:** В функциях `compute_damage` и `run_trajectory` отсутствует строгая проверка на отрицательные или нефизические значения `n` (число делений) и `t`. Добавление `assert!(n >= 0.0 && t_days >= 0.0)` улучшит надежность.
2.  **Документирование калибровки:** Значения по умолчанию в `CounterParams::default()` (`alpha=0.55`, `tau_days=32850.0`) выглядят как результат некоторой калибровки, но их происхождение не задокументировано в коде или соседних `README`. Следует добавить комментарии со ссылкой на метод калибровки (например, скрипт `calibrate.py`).
3.  **Явное указание на scaffold-статус:** В верхней части `lib.rs` и других ключевых файлов стоит добавить комментарий, что это реализация scaffold-проекта, где `gamma = 0` по умолчанию в соответствии с `CORRECTIONS_2026-04-22`, а параметры являются отправной точкой для будущей калибровки.

## 📋 Рекомендации

1.  **Привести `PARAMETERS.md` в соответствие с кодом и каноном:**
    *   Добавить строку для параметра `d_critical` со значением `0.55` (как в коде) и ссылкой на его использование.
    *   Для параметра `τ₂` (`tau_days`):
        *   **Либо** изменить статус с "TBD" на "Оценен (estimate)" и указать значение `~90 лет` (`32850 дней`) с пояснением, что это модель на основе анкеров калибровки.
        *   **Либо** обнулить значение по умолчанию в коде до `1.0` (как нейтральный элемент) и оставить статус "TBD", пока параметр не будет измерен.
    *   Добавить явный раздел "Нормализация", объясняющий, как литературные значения в bp преобразуются в безразмерные `alpha` и `beta` в коде (например, `alpha_code = (α_literature * n_star) / D₂,₀`).

2.  **Уточнить архитектуру связи между счетчиками:**
    *   **Вариант А (простой):** В `PARAMETERS.md` явно указать, что в текущей scaffold-реализации все гипотетические `Γ_{2,j}` свернуты в один общий параметр `gamma`.
    *   **Вариант Б (расширяемый):** Изменить структуру `CounterParams`, заменив поле `gamma: f64` на `coupling_coeffs: [f64; 4]` или `HashMap<CounterId, f64>`, чтобы отразить специфику связей с другими счетчиками (Centriolar=1, MitoROS=3, etc.). Это потребует правок в `compute_damage`.

3.  **Добавить проверки и документацию в код:**
    *   В `src/lib.rs`, в функцию `compute_damage`, добавить проверку неотрицательности `n` и `t_days`.
    *   В `impl Default for CounterParams` добавить комментарий: `// Default scaffold values. Gamma=0 per CORRECTIONS_2026-04-22. Alpha/Beta calibrated via scripts/calibrate.py.`
    *   В `src/tissue.rs` для метода `params()` добавить ссылку на то, что эвристики (`alpha *= 0.05` для нейронов) основаны на качественных утверждениях из `KNOWLEDGE.md`.

4.  **Синхронизировать скрипт калибровки:**
    *   В `scripts/calibrate.py` константы `N_STAR_DIV = 50.0` и `TAU_YEARS = 90.0` жестко закодированы. Следует сделать их входными аргументами или импортировать из `src/lib.rs`, чтобы избежать расхождений. Вывод скрипта (`PARAMETERS_calibrated.json`) должен автоматически обновлять значения по умолчанию в коде или, как минимум, сравниваться с ними.

**Общий вердикт:** Кодовая база Telomere в основном соответствует core-документам и соблюдает ключевые канонические исправления от 2026-04-22. Основные проблемы носят характер документационных расхождений и архитектурных упрощений, характерных для стадии scaffold-проекта. Рекомендованные правки направлены на устранение этих противоречий и повышение четкости проектирования.
